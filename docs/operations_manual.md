# 사후 운영 매뉴얼 + ROI 계산서

> **목적**: "이거 어떻게 돌리나" 평가위원 의심 즉시 해소 + 발전사 도입 결정 지원
> **분량**: 보고서 별첨 F (운영 매뉴얼) + 별첨 G (ROI)

---

## Part A. 운영 매뉴얼 (1주 기준)

### 일간 작업 (운영자 1명, 약 30분/일)

```
[09:00] 일간 자동 리포트 메일 확인 (자동 발송)
        - F1, Recall, Precision, Auto-Execute Ratio 점검
        - P1 알람 (자동 실행) 결과 확인
        - P2 알람 (검토 대기) 처리 → 운영자 승인/거부

[14:00] 오후 점검 (15분)
        - Streamlit 대시보드 확인
        - OTS < 60 알람 발생 시 원인 분석
        - 측정 불확도 평균 추세 확인

[필요 시] 임계 이벤트 대응
        - Calibration drift score > 0.7 → 측정기 점검 신청
        - Auto-Execute Ratio < 30% → XAI 일관성 점검
```

### 주간 작업 (월요일 09:00, 약 1시간)

```
1. 주간 리포트 자동 생성 확인
   - Coverage Rate (95%) 추세
   - Brier Score, RMS z-score
   - 운영자 피드백 (P2 알람 승인/거부 비율)

2. 모델 헬스 체크
   - SHAP 일관성 점수 (≥0.85 유지)
   - 신규 데이터로 ETL 파이프라인 정상 가동
   - 결측·이상치 비율 확인

3. 다음 주 점검 우선순위 결정
```

### 월간 작업 (1일 09:00, 약 2시간)

```
1. 월간 종합 리포트 작성
2. 운전 권고 채택률 분석 (목표 ≥50%)
3. 환경부 보고용 데이터 추출
4. 모델 재학습 필요성 평가
   - 데이터 분포 변화 (Concept Drift) 확인
   - 신규 패턴 발견 시 Type Taxonomy 업데이트
```

### 분기 작업 (3개월에 1회, 약 8시간)

```
1. 모델 재학습 (필요 시)
2. 측정기 정기 교정 + 데이터 기반 비교
3. CBAM 보고 정확도 검증
4. ISO 17025 외부 인증 준비 (연 1회)
```

## Part B. 시스템 아키텍처 운영 관점

```
[데이터 수집층]
  TMS 5분 → MQTT 브로커 → 시간당 집계 → InfluxDB
  AirKorea API → 1시간 폴링 → InfluxDB
  KMA API → 1시간 폴링 → InfluxDB
  KPX EPSIS → 1시간 폴링 → InfluxDB
            ↓
[ETL · 모델 추론층]
  4-source ETL → Feature Store → Model Inference (LSTM-AE + IForest)
                                  + GUM Uncertainty + SHAP
            ↓
[알람 · 의사결정층]
  Operator Trust Calibration → P1/P2/P3 등급화
  Slack/이메일 알람 송출
            ↓
[대시보드 · 리포트층]
  Streamlit (실시간) + 자동 리포트 (일/주/월/분기)
```

**기술 스택 (운영 환경)**:
- Python 3.10+ / PyTorch / scikit-learn
- InfluxDB (시계열 DB) / Redis (캐시)
- Streamlit (대시보드) / Grafana (모니터링)
- Docker Compose / Kubernetes (선택)
- Slack/Email (알람)

**서버 사양 (1호기 기준)**:
- CPU: 4 vCPU (Intel Xeon급)
- RAM: 16 GB
- Storage: 1 TB SSD (시계열 DB + 모델·로그)
- GPU: 선택 (CPU만으로도 5분 단위 추론 충분)
